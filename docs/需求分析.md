# 数字闯关 — 需求分析文档

**日期**: 2026-06-24  
**项目名称**: 数字闯关（Number Quest）  
**项目形态**: JavaFX 桌面应用 + 本地 JSON 存档  
**目标用户**: 6-9 岁小学生（1-3 年级）

---

## 一、项目背景

小学阶段是数学基础能力形成的关键期。传统口算练习形式单一，学生容易感到枯燥。通过将算术练习与冒险游戏结合，利用回合制答题战斗机制激发学习动机，让学生在“玩”中自然巩固加减乘除运算能力。

本项目以桌面游戏形式实现，使用 JavaFX 负责界面交互，使用本地 JSON 保存存档与资源配置，避免引入额外后端服务，降低运行门槛。

---

## 二、用户画像

| 角色 | 特征 | 核心诉求 |
|------|------|---------|
| 小学生（玩家） | 6-9岁，识字量有限，注意力集中约15-20分钟 | 好玩、看得懂、有成就感 |
| 家长 | 关注学习效果 | 能直观看到孩子练习了哪些内容、正确率如何 |
| 教师（可选） | 需要了解学生薄弱点 | 了解基础答题与闯关进度 |

首版以**小学生**为核心用户，优先保证游戏性和可理解性。

## 四、补充需求与边界条件

### 4.1 题目与难度

- 题目类型包含整数加减乘除。
- 题目难度按关卡配置，随着关卡提升题目范围扩大。
- 普通模式倒计时 10 秒，困难模式倒计时 3 秒。

### 4.2 战斗规则

- HP 会跨关卡继承，直到归零失败。
- 战斗中只保留基础答题与普通攻击反馈，不再包含独立道具系统。

### 4.3 存档与进度

- 新游戏需要重新初始化进度。
- 继续游戏必须在存档存在时可用；无存档时应提示并引导新游戏。
- 战斗中途退出后，重新进入应尽量恢复到最近一次存档状态。

### 4.4 资源配置

- 关卡、怪物配置使用 JSON 文件维护，降低改动成本。
- 资源缺失时应有默认占位图或降级提示，避免程序直接崩溃。

## 五、非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 界面切换 ≤ 0.5s，答题响应 ≤ 0.1s |
| 可用性 | 按钮和文字足够大，关键操作有明确反馈 |
| 可靠性 | 存档写入失败时不影响当前游戏运行 |
| 可维护性 | 关卡/怪物配置用 JSON 文件，修改数值无需改代码 |
| 兼容性 | Windows 10/11，JDK 17+ |

## 六、界面原型简描

### 主菜单
- 顶部：游戏标题
- 中央：两个大按钮“开始游戏”、“继续游戏”

### 地图界面
- 顶部：玩家状态栏
- 中央：关卡节点区域，显示当前关卡与 BOSS 节点
- 下方：普通 / 困难难度切换、返回标题

### 战斗界面
- 左侧：玩家角色像素图 + HP条
- 右侧：怪物像素图 + HP条
- 中央：题目卡片和 4 个选项按钮
- 上下区域：倒计时条、反馈文字、逃跑按钮

### 结算界面
- 中央：胜利或失败结算结果
- 下方：下一关、复活、返回地图

## 七、数据流向

```text
本地 JSON 存档
    ↓↑ 读取 / 保存
SaveData / PlayerDao
    ↓
GameState（游戏状态）
    ↓
JavaFX Scene（UI 渲染）
```
