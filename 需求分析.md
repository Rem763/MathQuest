# 数字闯关 — 需求分析文档

**日期**: 2026-06-24  
**项目名称**: 数字闯关（Number Quest）  
**目标用户**: 6-9 岁小学生（1-3 年级）

---

## 一、项目背景

小学阶段是数学基础能力形成的关键期。传统口算练习形式单一，学生容易感到枯燥。通过将算术练习与冒险游戏结合，利用回合制答题战斗机制激发学习动机，让学生在“玩”中自然巩固加减乘除运算能力。

---

## 二、用户画像

| 角色 | 特征 | 核心诉求 |
|------|------|---------|
| 小学生（玩家） | 6-9岁，识字量有限，注意力集中约15-20分钟 | 好玩、看得懂、有成就感 |
| 家长 | 关注学习效果 | 能直观看到孩子练习了哪些内容、正确率如何 |
| 教师（可选） | 需要了解学生薄弱点 | 分知识点统计答题情况 |

首版以**小学生**为核心用户，优先保证游戏性和可理解性。

## 四、补充需求与边界条件

### 4.1 题目与难度

- 首版题目仅支持四则运算中的加减乘，除法可作为后续版本扩展。
- 题目难度按年级/关卡配置，配置文件中应可独立调整每关的出题范围。
- 单题答题时间为 3 秒，超时按答错处理。

### 4.2 战斗与道具

- 战斗中允许使用道具，但每次操作需要明确反馈是否生效。
- 技能点上限建议为 3 点，避免资源无限积累。
- 道具使用与技能释放应避免与答题计时互相冲突。

### 4.3 存档与进度

- 新游戏需要清空旧存档或提供覆盖确认，避免误覆盖。
- 继续游戏必须在存档存在时可用；无存档时应提示并引导新游戏。
- 战斗中途退出或程序异常关闭后，重新进入应尽量恢复到最近一次存档状态。

### 4.4 资源配置

- 关卡、怪物、道具配置使用 JSON 文件维护，降低改动成本。
- 资源缺失时应有默认占位图或降级提示，避免程序直接崩溃。

## 五、非功能需求

| 类别 | 要求 |
|------|------|
| 性能 | 界面切换 ≤ 0.5s，答题响应 ≤ 0.1s |
| 可用性 | 按钮和文字足够大（适配低龄操作），关键操作有确认弹窗 |
| 可靠性 | 战斗中途退出不丢数据，存档写入失败有重试机制 |
| 可维护性 | 关卡/怪物/道具配置用 JSON 文件，修改数值无需改代码 |
| 兼容性 | Windows 10/11，JDK 17+ |

## 六、界面原型简描

### 主菜单
- 顶部：游戏标题“数字大冒险”，丛林藤蔓装饰
- 中央：两个大按钮“新游戏”、“继续游戏”（带图标）
- 底部：设置齿轮图标（小按钮）

### 地图界面
- 背景：绿色丛林插画（深林/草地/溪流分层）
- 顶部：年级标签页 + 金币/HP 状态栏
- 中央：关卡图标沿曲线路径排列，锁定的灰色，未打的发光，打完的带星星
- 右上角：商店按钮（宝箱图标）

### 战斗界面
- 左侧：怪物立绘 + HP条 + 名称
- 右侧：玩家角色 + HP条 + 技能点指示
- 中央底部：题目卡片（大字号），4 个选项按钮（2×2 排列）
- 顶部栏：技能按钮 + 道具按钮 + 倒计时环

### 商店界面
- 顶部：金币余额
- 中央：道具网格（图标 + 名称 + 效果描述 + 价格标签）
- 底部：返回地图按钮

### 结算界面
- 中央：星级动画（1~3颗星依次点亮）
- 下方：金币奖励数字跳变动画
- 底部：两个按钮“下一关”、“返回地图”

## 七、数据流向

```text
MySQL (player/items/stars)
    ↓↑ 加载 / 写入
DatabaseManager → DAO → 业务逻辑
                         ↓
JSON配置 (levels/enemies/items) → 配置加载器 → 业务逻辑
                                                  ↓
                                          JavaFX Scene (UI渲染)
```
